# AX GROUP — AI Transformation eXperience

---

## 팀 선언

> **"시장이 방향을 정하기 전, AI•AX를 실제 업무에 선적용하여 우리만의 실행 우위를 만든다"**

우리는 대한민국 SI 에이전시 안에서 움직이는 실험 조직이다.
클라이언트의 요구를 받아 개발하는 것이 기존의 방식이라면,
우리는 **AI가 업무를 어떻게 바꿀 수 있는지를 먼저 증명하고, 그 경험을 전파하는 팀**이다.

---

## AX의 의미

| 기호 | 의미 |
|------|------|
| **A** | AI — 인공지능을 도구가 아닌 동료로 |
| **X** | eXperience — 실험하고 경험하며 증명 |
| **X** | 변수 (Variable) — 고정되지 않고 상황에 맞게 변한다 |
| **X** | 함수 (Function) — 입력(문제)을 받아 출력(가치)을 만든다 |

```
f(problem) = value
AX(현실의 문제) = AI로 가속된 실제 결과물
```

우리는 변수처럼 유연하게 움직이고, 함수처럼 명확한 결과를 만든다.

## 팀 철학 (Philosophy)

### 1. 선험(先驗) — 먼저 해본 사람이 말할 수 있다

AI 도구, 자동화 워크플로우, 새로운 개발 패러다임.
우리는 회사가 "검토해봐" 라고 하기 전에 이미 써보고 있어야 한다.
경험 없는 의견은 의견이 아니다. **직접 부딪혀 얻은 인사이트만이 조직을 설득할 수 있다.**

### 2. 실행 우선 (Execution First)

완벽한 계획보다 불완전한 실행이 낫다.
MVP를 만들어 보여주고, 작동하는 것으로 대화한다.
**"이런 게 가능합니다"가 아니라 "이게 작동합니다"로 말한다.**

### 3. 조용한 영향력 (Silent Influence)

우리의 목표는 팀 내 주목이 아니다.
프로젝트 팀이 우리와 일한 뒤 "이 방식으로 하면 더 빠르네"라고 느끼게 만드는 것.
**소음 없이 기준을 바꾸는 것이 진짜 영향력이다.**

### 4. AI는 대체재가 아니라 증폭기

AI가 나를 대신하는 것이 두렵다면, AI를 쓰는 사람에게 자리를 빼앗길 것이다.
우리는 AI를 쓰는 법을 배우는 팀이 아니다.
**AI와 함께 이전에는 불가능했던 것을 만드는 팀이다.**

### 5. AI는 파트너다 — 그러나 판단은 내가 한다

AI는 강력한 파트너이지만, 주객이 전도되어서는 안 된다.
많은 사람들이 사실 확인을 넘어 자신의 주장과 신념까지 AI의 말로 대체하는 것을 본다.
그것은 생각을 빌려쓰는 것이고, 결국 스스로 판단하는 능력을 잃는 것이다.

AI가 답을 내놓더라도, **그 답이 맞는지 틀린지를 판단하는 것은 여전히 우리의 몫이다.**
우리는 AI를 잘 쓰는 팀인 동시에, **AI에 기대지 않고도 논리적으로 사고하고 결정할 수 있는 팀**이어야 한다.

> *"AI의 출력은 참고 자료다. 내 결론은 내 사고에서 나와야 한다."*

### 6. 문서화는 자산이다

경험은 공유되어야 자산이 된다.
우리가 시도한 것, 실패한 것, 성공한 것 — 모두 기록한다.
**팀의 지식은 개인의 머릿속이 아닌 공유된 공간에 있어야 한다.**

## 철칙 (Operating Principles)

### 일하는 방식

- **실제 업무에 적용하지 않은 것은 배운 것이 아니다.**
  세미나, 아티클, 유튜브로 알게 된 것은 '정보'일 뿐. 실제 프로젝트에 써봐야 '경험'이 된다.

- **작게 시작하고, 빠르게 검증하고, 담대하게 확장한다.**
  큰 변화는 작은 실험의 반복에서 온다.

- **도구는 수단이고, 목적은 결과다.**
  Claude를 쓰든 Cursor를 쓰든 상관없다. 더 나은 결과물을 더 빨리 만들면 된다.

- **혼자 막히면 24시간 안에 물어본다.**
  고민은 미덕이 아니다. 공유가 속도를 만든다.

### 문화

- **비판보다 대안.** 안 된다고 말할 때는 반드시 "이렇게 하면 어떨까요?"를 함께.

- **실패는 기록하고 공유한다.** 같은 실패를 두 번 하지 않기 위해.

- **서로의 성장에 투자한다.** 내가 아는 것을 나누는 것이 팀의 수준을 올린다.

- **레거시를 존중하되, 레거시에 갇히지 않는다.**
  기존 방식이 통했던 이유를 이해하되, 더 나은 방법이 있다면 제안한다.

- **안 즐겁게 일은 못하되, 일을 일로서 대하자.**
  펼쳐진 퍼즐 조각이 많은 만큼 힘들고 지칠 때가 있다. 그건 당연하다.
  하지만 회사 밖에서도 쌓이는 게 있는데, 일할 때만큼은 서로를 이해하고 배려하며 같이 재밌게 하자.
  즐기면서 하는 일이 결국 더 좋은 결과를 만든다.

## 실행 방안 (Execution Roadmap, 2026년 2월 기준)

> *선언은 실행이 없으면 포스터다. 아래는 지금 당장 시작할 수 있는 것들이다.*

### Phase 0 — Quick Win (지금 바로)

**목표**: 작동하는 것 하나를 보여준다. 설득은 데모로 한다.

#### 1. AI 코드리뷰 파이프라인 구축

```
PR 생성 → GitHub Actions 트리거 → Claude API 호출
        → 코드 품질 / 보안 / 접근성 피드백 → PR 코멘트 자동 등록
```

- SI 프로젝트 특성상 QA·검수 비용이 큰 구간 → AI가 1차 필터
- **측정 지표**: PR 당 리뷰 소요 시간 Before/After 비교
- **도구**: GitHub Actions + Claude API (`claude-sonnet-4-6`)

#### 2. 팀 프롬프트 라이브러리

- Notion 또는 GitHub Wiki에 "검증된 프롬프트" 공유 공간 개설
- 형식: `상황 | 프롬프트 | 결과 | 검증자`
- **첫 달 목표**: 각자 3개씩 → 팀 라이브러리 15개 이상
- 반복되는 요청(설계서 초안, 메일 작성, 테스트케이스 생성)부터 수집

#### 3. Claude Code 페어 프로그래밍 온보딩

- 팀원 1명씩 Claude Code 셋업 + 30분 페어 세션 진행
- 목표: 현재 진행 중인 SI 업무 하나에 실제 적용
- **결과 공유**: Slack/노션에 "써봤더니 이랬다" 짧게라도 기록

---

### Phase 1 — 내재화 (1~3개월)

**목표**: 개인 도구 사용 → 팀 워크플로우로 전환

#### 4. SI 산출물 자동화 — 반복 문서의 AI 초안화

SI 업계의 가장 큰 비용 중 하나는 **문서 작업**이다.

| 산출물 | AI 적용 방식 |
|--------|-------------|
| 화면 설계서 초안 | 요구사항 입력 → 섹션·항목 자동 구성 |
| 테스트케이스 | 기능 명세 → 시나리오 자동 생성 |
| API 명세서 | 코드/기획 → OpenAPI YAML 초안 |
| 회의록 요약 | 녹취/메모 → 액션아이템 구조화 |

- **방식**: n8n 자동화 워크플로우 또는 Claude API 직접 연동
- **기준**: 초안 완성도 70% 이상이면 성공 — 수정이 창작보다 빠르다

#### 5. 월 1회 AX 데모데이 (내부)

- 형식: 발표 10분 + Q&A 5분, 팀원 순번제
- 주제: "이번 달에 AI로 해본 것 — 뭘 했고, 어떻게 됐고, 다음엔 뭘 할 것인가"
- **기록**: 발표 자료 + 핵심 인사이트를 팀 Notion에 누적
- 목적: 실험 문화 형성 + 암묵지를 명시지로

#### 6. AI 활용 수준 자가 진단 — 팀 기준선 만들기

```typescript
type AIUsageLevel = {
  0: "아직 안 씀";
  1: "가끔 ChatGPT로 검색 대체";
  2: "코드·문서 작업에 정기적으로 활용";
  3: "워크플로우에 AI가 포함됨 (자동화 일부)";
  4: "AI 없이는 현재 속도가 불가능한 수준";
};
```

- 팀 전체 현재 레벨 파악 → 목표 레벨 설정 → 3개월 후 재측정
- **기준**: 남과 비교가 아닌, 내가 3개월 전의 나보다 나아졌는가

### Phase 2 — 확장 (3~6개월)

**목표**: 팀 안에서 증명한 것을 조직 밖으로 전파

#### 7. 사내 AX 가이드 발행 — "우리가 해봤다"

- 타 팀이 "AI 써보고 싶은데 어디서 시작하지?"라고 물을 때 줄 수 있는 문서
- 내용: 추천 도구 + 실제 적용 사례 + 주의사항 (환각, 보안, 저작권)
- **형식**: 사내 위키 또는 PDF 한 장짜리 Quick Start

#### 8. 클라이언트 프로젝트에 AI 기능 1개 제안

- 기존 SI 수주 프로젝트에 "AI 연동 옵션" 제안
- 예시: 챗봇 FAQ 자동화, 공문서 요약, 검색 고도화 (Semantic Search)
- **목표**: 제안서 통과 1건 — AX팀이 SI 수익에 기여하는 첫 사례

#### 9. AX 실험 로그 공개 — 조직 신뢰 자산 구축

- 팀 내부에서 쌓은 실험 기록을 사내 공개 채널에 월 1회 공유
- "이렇게 해봤더니 30% 빨라졌다" 같은 수치 기반 사례
- 목적: AX팀 = "실제로 해본 팀"이라는 인식 정착

## 우리가 함께 갈 미래 — 논의 주제

> *이번 자리는 선언이 아니라 대화다. 우리가 어떤 미래를 만들고 싶은지 함께 이야기하자.*

### 중기 방향 (6개월~1년)

- **AI 활용 내재화**: 개인 차원의 툴 사용을 → 팀 단위 워크플로우로 전환
  - 반복 업무 자동화 (보고서, QA, 코드리뷰)
  - 프롬프트 라이브러리 공유 및 표준화

- **AX 실험 캐릭터 구축**: 프로젝트 외 시간에 자체 실험 진행
  - 매달 1개씩 "이걸 AI로 해봤더니..." 내부 공유

- **조직 내 AX 전도사**: 다른 팀이 물어볼 때 답할 수 있는 팀

### 함께 논의하고 싶은 것

1. **우리 팀의 정체성을 어떻게 규정하고 싶나요?**
   "뭐든 잘 만드는 팀" vs "AI로 일하는 방식을 바꾸는 팀"

2. **각자가 생각하는 AX 팀의 1년 후 모습은?**

3. **지금 당장 AI를 적용해보고 싶은 업무/문제가 있나요?**

4. **팀 안에서 각자의 역할/강점을 어떻게 활용하면 좋을까요?**

5. **이런 팀이 됐으면 좋겠다 — 각자의 한 문장**

## 코딩본 — 자신을 표현하는 방식

> PPT 3장 대신, **현재의 나를 코드로 표현한다.**
> 언어, 형식, 길이는 자유. 단, 진짜 나여야 한다.

```typescript
// 예시: 조경문의 코딩본
const me = {
  name: "조경문",
  role: "Frontend Developer",

  currentState: {
    focus: ["AI-assisted development", "DX 개선", "인터랙션 설계"],
    learning: ["Agent 기반 워크플로우", "LLM 프롬프트 최적화"],
    using: ["Claude Code", "n8n", "GSAP", "Three.js"]
  },

  belief: "좋은 코드는 사람을 위한 경험을 만든다",

  question: "AI와 함께 일할 때, 나는 어떤 사람이 되고 싶은가?",

  // 이번 발표에서 하고 싶은 말
  message: "우리가 함께 실험하면, 더 빨리, 더 멀리 갈 수 있다"
};
```

**각자의 코딩본이 담아야 할 것:**
- 지금 내가 집중하는 것
- 내가 팀에 기여할 수 있는 것
- 이 팀에서 해보고 싶은 것 또는 던지고 싶은 질문
